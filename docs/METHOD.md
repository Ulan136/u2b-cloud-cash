# МЕТОД: как строить любой веб‑проект

Универсальное руководство и порядок действий для нового веб‑приложения. Правила ниже —
общие для любого домена (склад, CRM, запись к врачу, что угодно). Примеры из репозитория
`u2b-cloud-cash` даны **только как иллюстрации** — копируйте метод, а не касу.

Пиши код так, как написан код вокруг: одна ответственность на файл, один слой не лезет в
чужой. Проверка после каждого шага важнее скорости.

---

## 1. Стек и почему

| Слой | Технология | Почему |
|---|---|---|
| Фронт+бэк | **Next.js 14, App Router, TypeScript** | Одно приложение: страницы (View) и API (`app/api/*/route.ts`) в одном репозитории и деплое. TS ловит ошибки до рантайма. |
| Стили | **Tailwind CSS** | Быстрая вёрстка утилитами, без отдельных CSS‑файлов; тему держим на CSS‑переменных в `globals.css`. |
| БД | **Neon PostgreSQL (serverless)** | Бесплатный тариф, HTTP‑драйвер работает из serverless‑функций, автоскейл до нуля. |
| ORM | **Drizzle ORM** | Схема в TypeScript, типобезопасные запросы, `drizzle-kit push` синхронизирует БД без ручных миграций на старте. |
| Хостинг | **Vercel** (автодеплой из GitHub) | Пуш в `main` → сборка и деплой. Бесплатный Hobby‑тариф. |
| Установка на телефон | **PWA** (manifest + service worker) | «Добавить на главный экран», полноэкранный режим. |

**Что бесплатно и лимиты (важно заложить в архитектуру сразу):**
- **Vercel Hobby:** serverless‑функции без вечных процессов (нельзя держать сокет/таймер в памяти между запросами); **cron только раз в сутки** — ежечасный/ежеминутный cron **молча роняет ВСЕ сборки** (см. Грабли). Планируйте фоновые задачи под 1 запуск/сутки, а «частоту» реализуйте логикой внутри роута.
- **Neon Free:** есть лимит на хранилище/вычисления и «засыпание» БД; для веб‑serverless используйте **pooler‑строку подключения** (host с `-pooler`), драйвер `@neondatabase/serverless` (HTTP).
- **Serverless = без состояния:** никаких «живых» WebSocket‑серверов в приложении. Реалтайм — через внешний сервис (напр. Pusher) или polling.

## 2. Архитектура‑шаблон (для любого домена)

Строгие слои, поток данных сверху вниз:

```
HTTP → app/api/<раздел>/route.ts   (Controller — тонкий: валидирует DTO, зовёт service, отдаёт JSON)
          → dto/<раздел>.dto.ts     (zod‑схемы входных данных)
          → services/<раздел>.service.ts  (ВСЯ бизнес‑логика: формулы, оркестрация, транзакции; про HTTP не знает)
              → repositories/<сущность>.repo.ts  (ТОЛЬКО запросы Drizzle: find/create/update/delete/агрегаты)
                  → db/schema.ts     (Entity: таблицы Drizzle — единственный источник структуры)

lib/            — общее: подключение к БД, форматирование, живое обновление, ошибки
app/<раздел>/page.tsx — View: только отображение и вызовы API (fetch)
```

**Зачем именно так:** бизнес‑правило живёт в одном месте (service). Формулу меняешь один раз, и
её видят все страницы и агрегации. Пример‑иллюстрация из репозитория: расчёт «ОБЩ РЕАЛ / МИН/ПЛЮС»
лежит чистыми функциями в `services/kassa.service.ts` и переиспользуется в `services/reports.service.ts`
и `services/dashboard.service.ts` — ни одной копии формулы.

**Правила‑запреты (нарушение = технический долг):**
- **SQL/запросов Drizzle в контроллерах — нельзя.** Контроллер только: распарсить запрос → `schema.safeParse` → вызвать service → `NextResponse.json`.
- **Бизнес‑логики во View — нельзя.** Страница не считает итоги/формулы, только показывает то, что вернул API.
- **Дублировать формулу/агрегацию — нельзя.** Вынеси в service и переиспользуй.
- **Один тонкий контроллер повторяется** — заведи фабрику (в репозитории `lib/directoryRoute.ts` собирает GET/POST/PATCH для однотипных справочников).
- **Транзакции:** HTTP‑драйвер Neon не поддерживает интерактивные транзакции Drizzle. Для атомарности нескольких запросов бери «сырой» neon‑клиент и батч: `sqlClient.transaction([q1, q2])` (в репозитории так сделано каскадное переименование).

**Живое обновление (заложить сразу, реалтайм — потом):** транспорт‑слой с интерфейсом
(`subscribe(channel, onSignal)`), сейчас единственная реализация — polling по таймеру. Хук
`useLiveData(channel, load, deps)` перезагружает данные по сигналу и мгновенно при возврате
фокуса/сети. Позже подменяешь polling на Pusher **одной фабрикой** — страницы и хук не меняются.

## 3. Порядок построения с нуля (каждый шаг — проверка перед следующим)

1. **GitHub repo (private).** Создать пустой приватный репозиторий.
2. **Локальная папка + git.** `git init`, привязать remote.
3. **Neon проект → `DATABASE_URL`.** Создать БД, скопировать **pooler**‑строку (host с `-pooler`, `?sslmode=require`). Никуда не коммитить.
4. **`create-next-app` в ПОДПАПКЕ `nextjs`** (не в корне!). Флаг `--src-dir` обязателен:
   `npx create-next-app@14 nextjs --typescript --tailwind --app --src-dir --import-alias "@/*"`
   Подпапка нужна, чтобы в корне лежали `docs/`, скрипты, а Vercel Root Directory = `nextjs`.
5. **Зависимости:** `@neondatabase/serverless drizzle-orm zod`; dev: `drizzle-kit tsx`.
6. **`drizzle.config.ts` + `src/db/`:** `schema.ts` (таблицы), `lib/db.ts` (подключение: экспорт `db` и «сырого» `sqlClient`). В `drizzle.config.ts` вручную подгрузи `.env.local` — `drizzle-kit` его сам не читает.
7. **`.gitignore` проверить ДО первого коммита:** `.env`, `.env*.local`, `node_modules/`, `.next/`, `drizzle/`, `*.log`, файлы‑секреты.
8. **`.env.local` (`DATABASE_URL`) → `npx drizzle-kit push`.** Проверить, что таблицы создались.
9. **Страницы по одной, с главной бизнес‑страницы.** После КАЖДОЙ страницы/раздела: `npx tsc --noEmit` → `npm run build` → smoke‑тест API (реальный запрос к роуту на dev‑сервере) → `commit` → `push`. Не копи несколько разделов в одном коммите.
10. **Живое обновление:** хук `useLiveData` + транспорт‑слой (polling сейчас, интерфейс под real‑time на будущее). Фон обновляет только просмотровые данные, форму пользователя не затирает.
11. **PWA:** `public/manifest.json` (name, `display:standalone`, `theme_color`, иконки 192/512), регистрация service worker, `metadata.manifest` в layout.
12. **Vercel:** Import репозитория → **Root Directory = `nextjs`** → добавить env‑переменные (`DATABASE_URL` и пр.) **ДО** первого деплоя.
13. **Cron при необходимости** — только суточный (`0 H * * *`), защита секретом (`Authorization: Bearer <CRON_SECRET>`), «частоту» внутри часа реализуй логикой роута.

## 4. Рабочий процесс с ИИ‑терминалом

- **Одна задача за раз.** Один раздел/фича на промпт — легче проверять и откатывать.
- **Каждый промпт заканчивается:** «`npx tsc --noEmit`, `npm run build`, smoke‑тест API, `commit` + `push`». Без этого «done» не считается.
- **Боевая БД:** тестовые записи — точечно (уникальные даты/имена вроде `2031-…`, `ZZ …`), удалять **только свои** строки, никаких массовых `DELETE`/`TRUNCATE`. **Перед массовыми операциями — бэкап** всей базы в JSON.
- **Длинные промпты/файлы:** проверяй, что вставились целиком (grep по ключевым строкам), прежде чем запускать.
- **После каждого `push` — проверка на проде** (открыть URL, `curl` ключевой страницы). Пуш ≠ «увидели пользователи».
- **Не удаляй то, что не создавал:** увидел незнакомые строки в БД — сначала посмотри (`SELECT`), это могут быть реальные данные.

## 5. ⚠️ Грабли (проверено на этом проекте, применимо к любому)

- **`app/` в корне конфликтует с `src/app/`.** Флаг `--src-dir` обязателен; в `tsconfig.json` пути `"@/*": ["./src/*"]`. Иначе Next не находит страницы или дублирует роутинг.
- **Секреты — в `.gitignore` ДО первого коммита.** Перед каждым push: `git diff --cached --name-only` и грепом убедиться, что `.env*`/секреты/большие исходники не попали в коммит.
- **Vercel Hobby: ежечасный/ежеминутный cron роняет ВСЕ сборки молча** (`deploy_failed: Hobby accounts are limited to daily cron jobs`). Прод застревает на последнем удачном билде, а новые пуши «не выходят». Разрешено только `"0 H * * *"` (раз в сутки).
- **Redeploy/Rollback в панели Vercel закрепляют старую сборку** (staging/rollback pin). Обычное обновление — это `git push`. Застрявший Production лечится через CLI: `vercel link` (из **корня репо**, т.к. Root Directory = `nextjs`) → `vercel ls` → `vercel promote <url>` или `vercel deploy --prod` из корня.
- **Имена папок страниц и `href` в меню — сверять.** Папка `app/reports/` но ссылка `/otchety` = 404 на проде. Держи список пунктов меню и проверяй соответствие путям.
- **Кириллица/UTF‑8 — roundtrip‑тест.** Инструменты в терминале Windows могут ломать кодировку: проверяй сохранение→чтение через UTF‑8‑клиент (node fetch), а не через `curl` в консоли.
- **Excel‑даты = серийные числа от эпохи 1899‑12‑30** (`new Date(Math.round((n-25569)*86400*1000))`). При импорте: **вычисляемые/автополя не импортировать** — они считаются в сервисе; тащить только ручной ввод.
- **Service worker и «старая версия».** Если добавляешь оффлайн‑кеш — только **network‑first**, иначе пользователи застревают на прошлой сборке. В этом проекте SW **сознательно не кеширует** (пустой `fetch`‑обработчик, только для установки PWA) — именно чтобы не показывать старьё.
- **Деньги — `numeric`, не float.** Пустые/нулевые значения нормализуй к `"0"`. Числа моноширинно с разделителями.
- **Архивация вместо удаления.** Флаг `archived` — история операций сохраняется, но сущность скрыта из выпадающих списков. Никогда не теряй операции.
- **Unique — на естественных ключах** (напр. имя работника, `unique(year, month)`), это ловит дубли на уровне БД.
- **Переименование, на которое ссылаются по имени** (не по id) — делай каскадом в транзакции, иначе рассинхрон.

## 6. Стартовый чек‑лист нового проекта

- [ ] Приватный GitHub‑repo создан, локально `git init` + remote.
- [ ] Neon: проект создан, **pooler** `DATABASE_URL` скопирован.
- [ ] `create-next-app@14 … --src-dir --import-alias "@/*"` в подпапке `nextjs`.
- [ ] Установлены `@neondatabase/serverless drizzle-orm zod` (+ dev `drizzle-kit tsx`).
- [ ] `drizzle.config.ts` (грузит `.env.local`), `src/db/schema.ts`, `src/lib/db.ts` (`db` + `sqlClient`).
- [ ] `.gitignore` закрывает `.env*` и секреты — **проверено до первого коммита**.
- [ ] `.env.local` заполнен → `drizzle-kit push` → таблицы на месте.
- [ ] Заготовлены слои: `dto/`, `services/`, `repositories/`, `lib/`. Контроллеры тонкие.
- [ ] Первая бизнес‑страница + её API; после — `tsc --noEmit`, `build`, smoke‑тест, commit, push.
- [ ] `useLiveData` + транспорт‑слой (polling, интерфейс под будущий real‑time).
- [ ] PWA: manifest + иконки + регистрация SW (без агрессивного кеша).
- [ ] Vercel: Import, **Root Directory = `nextjs`**, env‑переменные добавлены ДО деплоя.
- [ ] Cron (если нужен) — суточный + защита `CRON_SECRET`.
- [ ] После деплоя: открыть прод‑URL и проверить, что видна свежая версия.

> Правило одной строкой: **тонкие контроллеры, вся логика в services, запросы в repositories,
> страницы только показывают; после каждого шага — tsc + build + тест + push; на проде — проверять.**
